# Task engine reads `.vscode/tasks.json` faithfully; dispatchers are thin adapters

> **Relocated 2026-07-03.** The engine and launcher now live in this repo at
> `scripts/task-engine/` and `scripts/launcher/` (adopted from
> `forwood-one-tools`, ADR numbering carried over). The distribution rationale
> in *Why TS at this path* below described the old home; the packages are no
> longer synced via personal-tooling — consumers use this checkout directly.
> Task *definitions* (each clone's `.vscode/tasks.json`) still ship with the
> product repos.

The product family ships a per-clone `.vscode/tasks.json` (e.g. `linked-tools/repos/forwood-one/.vscode/tasks.json`) that defines the dev-server lifecycle — API, WEB, SB, ROOTDB, "Kill all", "Branch change setup", "Spin up dev servers" — with full VS Code task semantics (`dependsOn`, `dependsOrder`, `presentation.panel`, `presentation.group`, `inputs`). VS Code itself surfaces these through its tasks panel; in a tmux-driven flow, the legacy bash `task-emit` + `forwood-task-picker` reads the same JSON and dispatches via `tmux send-keys` + `devcontainer exec`. We're now adding a maiTerm-native dispatcher (alongside an Ink launcher) and want a single substrate that both tmux and maiTerm sit on top of, with room for future surfaces (a VS Code-task-runner replacement, an SSH-attached remote launcher) without duplicating the JSON parser.

**Decision.** A new TS package at `linked-tools/core/scripts/task-engine/` is the single reader of `.vscode/tasks.json`. It emits **declarative task records that mirror VS Code's tasks.json shape unchanged** — `{ label, type, command, args, options: { cwd, env }, dependsOn, dependsOrder, presentation, isBackground, ... }` — and resolves VS Code variables (`${workspaceFolder}`, `${env:…}`, `${input:…}`). Dispatchers are thin adapters that translate records into native idioms: the tmux dispatcher composes `tmux send-keys … "devcontainer exec … <cmd>"`; the maiTerm dispatcher calls the maiTerm MCP server (`openTab` for fresh tabs, `sendKeysToTab` for VS Code's "re-run in dedicated panel" semantics on `presentation.panel: dedicated` + matching `presentation.group`). `task-emit` and its fixture tests retire; `forwood-task-picker` becomes a shell wrapper over the TS engine. Kill is not a per-tab MCP call — `.vscode/tasks.json` already models "Kill all" as a normal task that runs `kill-dev-ports.sh` (port-based via `lsof -ti :PORT | xargs kill -9`), and composite "Kill → Setup → Up" already runs through `dependsOn`; both flow through the engine and dispatcher unchanged.

**Why pass-through, not normalised.** The directive is "behave like VS Code's task runner". The cleanest expression is "emit VS Code's records unmodified" — `presentation.panel === 'dedicated' && presentation.group === 'API'` becomes the maiTerm dispatcher's signal to find-or-spawn a tab named `API`. Inventing our own `target` / `targetName` fields buys nicer naming at the cost of an opinionated translation layer the engine then has to maintain forever, and risks diverging from future VS Code schema additions. Dispatchers are the right home for translation; engine output should match what was authored.

**Why TS at this path.** TS gives the engine a real type-safe model of the VS Code schema (vital once `dependsOn` graphs and `${input:…}` variables come into play); Node cold-start (~150–300ms) is acceptable for both the one-shot Ink launcher and the rare-fire tmux picker keybinding. Bash + jq (today's `task-emit`) is adequate for flat tasks but becomes brittle once we honour `dependsOn` / variable resolution. Go was considered for faster tmux cold-start but introduces a new toolchain into the tools repo for marginal benefit. Homing the engine under `linked-tools/core/scripts/` lets `personal-tooling-sync` distribute it into consumer clones via the existing symlink bridge — the host launcher and the in-clone tmux picker both invoke the *same* node module via the synced path.

**Why MCP for the maiTerm dispatcher.** maiTerm already ships an authenticated MCP server (lockfile at `~/.claude/ide/<port>.lock` carries port + 32-char token; auth header `x-claude-code-ide-authorization`). A Node process is an MCP client, not necessarily a Claude agent — the launcher reads the lockfile, opens a WS connection, and calls tools. No new REST surface, no URL scheme, no sidecar CLI. The only missing pieces are two MCP tools (`openTab`, `sendKeysToTab`); these are being prototyped on a fork of `Flexmark-Intl/maiterm` and will be split into per-tool upstream issues with the fork branch linked as a serving suggestion. External-caller gates were verified (recon, see Tab struct already exposing `name: String` in `src-tauri/src/state/workspace.rs`): no CORS, no client-identity restriction, auth is solely the token, and the tools are global rather than tab-affinity-bound — so an external Node launcher can connect without a tab session.

**Launcher shape.** A one-shot Node + Ink CLI bound to a key combo, full-screen alt-buffer, **host-side** (the tab running the launcher is a host shell — not a devcontainer shell — so the lockfile is natively readable and the dispatcher composes `devcontainer exec` into the spawned task commands the same way `task-emit` does today). Lifetime: cold-start → menu → fire → exit. Zero background cost. Clone selection uses env var (`FORWOOD_CLONE`, set per maiTerm workspace) with a picker fallback when unset and `--clone` override.

**Consequences.** (1) Adding a future dispatcher (VS Code task panel native, SSH-remote launcher, web surface) is a single new adapter — engine and dispatchers stay independent. (2) Future VS Code task fields (e.g. `runOptions.instanceLimit`) flow through the engine without code changes; dispatchers opt in when they want to honour them. (3) The maiTerm fork carries two new MCP tools that we'll upstream; until they merge, the maiTerm dispatcher requires the fork build. (4) `task-emit` and its fixture tests are deleted in the same change that ships the TS engine and the rewritten tmux picker — no compatibility shim.
