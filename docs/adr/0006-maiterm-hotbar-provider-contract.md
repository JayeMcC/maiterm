# maiTerm hotbar: generic provider contract, host-default execution context, container gate

VS Code/Cursor's last irreplaceable role in the local dev flow is container handling: the tasks panel, Reopen/Rebuild in Container (the only bring-up path — `require-devcontainer.sh` merely guards), and the ports panel for ad-hoc forwards. ADR 0005 landed the substrate (task engine as single reader of `.vscode/tasks.json`, thin dispatchers, one-shot Ink launcher firing through the maiTerm MCP tools on the `JayeMcC/maiterm` fork). This ADR adds the surface that retires the VS Code dependency: the **hotbar**, a fold-out rail docked to maiTerm's right edge whose contents derive from the active tab's cwd.

**Decision — rail shape and detectors.** One rail, contextual to the active tab, folded away entirely when nothing is detected. Cheap native **detectors** walk up from the tab's cwd for marker files and each contributes a section: `.vscode/tasks.json` → task section, `.devcontainer/devcontainer.json` → container section. Detection controls visibility; live state (container up/down) controls enablement inside a section. The hotbar is a picker, not an executor: firing a task routes through the same maiTerm dispatcher as every other launcher surface (find-or-spawn dedicated tabs per `presentation`), and the invoking tab's cwd is a filter, never a target. Rejected: per-tab dropdown chrome (no home for port state), run-in-place execution (breaks `presentation` and `dependsOn: parallel` semantics).

**Decision — generic provider contract, forwood logic in the launcher.** The fork feature is forwood-agnostic and upstreamable as a "contextual action rail": it is configured with marker filenames plus external commands, and carries zero forwood strings. The task section runs a *list command* lazily on unfold (`forwood-launcher --list --json --dir <cwd>`, ~300ms node cold-start accepted) and a *fire command* per click (`forwood-launcher --fire <label> --dir <cwd>`), which dispatches via MCP loopback into the very maiTerm that spawned it — `openTab`/`sendKeysToTab` reused wholesale. The container section runs a *status command* (`forwood-launcher --container-status --dir <cwd>`), polled only while the rail is unfolded. The launcher gains dir-based resolution (walk up from `--dir` to the repo root as `${workspaceFolder}`), sidestepping `$FORWOOD_CLONE` for hotbar-initiated fires. Rejected: a fork-native `tasks.json` parser (second reader — exactly the divergence ADR 0005 forbids) and a resident forwood daemon (contradicts the zero-background-cost launcher philosophy). The `(markers, list, fire, status)` configuration is the upstream API and must stay stable once merged.

**Decision — execution context is host by default; the container gate signals otherwise.** Tasks run on the **host** unless their `dependsOn` chain (directly or through a composite) contains the well-known **container gate** task `Require devcontainer` — then the dispatcher composes `devcontainer exec`. No schema extension: a `runOn` field was considered and rejected because the signal already exists in the task graph, keeping ADR 0005's pass-through purity and full VS Code compatibility. The gate task becomes context-adaptive: inside a container it verifies and exits 0 (today's guard, VS Code behaviour unchanged); on the host it runs `devcontainer up --workspace-folder .` (idempotent bring-up) — one task, three roles: guard, context signal, bring-up. Consequently `Spin up dev servers` becomes the cold-machine "Reopen in Container" replacement on every surface, including tmux. Gate edges are written explicitly into `tasks.json` (each container-context leaf gains `dependsOn: ["Require devcontainer"]`) rather than kept as a config list in the launcher — the graph is the single source of truth, and VS Code fires improve (loud guard failure instead of confusing task death on a host window). New host-context tasks `Rebuild container` / `Rebuild container (no cache)` wrap `devcontainer up --remove-existing-container [--build-no-cache]`.

**Decision — ports are live truth; ad-hoc forwards are docker-owned sidecars.** The container section shows the *actual* published ports read off the running compose project (no `.env` parsing — live mappings already reflect per-clone overrides), each with service label, listening dot, and click-to-open. Standing ports need no forwarding — compose already publishes them; VS Code's `forwardPorts` list is locally redundant. What compose cannot cover is ad-hoc ports (an agent starts `:8080` inside the container; Docker cannot add published ports to a running container), so "Forward" spawns a labelled socat **sidecar** (`docker run --rm -d -p N:N --network <compose-network> alpine/socat … TCP:<dev-ip>:N` — *corrected at implementation: Docker refuses `-p` combined with `--network container:<id>`, so the sidecar joins the compose network and relays to the dev container's IP; see PLAN-15 spec-changelog Correction 3*); stop kills it. Sidecars are owned by Docker, not maiTerm — they survive terminal restarts and are enumerated by label. Unpublished container-side listeners are auto-detected (`docker exec … ss -tlnp` in the status poll) and offered for forwarding, with a manual port input as fallback. Rejected: a static expectation list parsed from config (lies when down or overridden).

**Consequences.** (1) The launcher package grows three non-interactive modes (`--list --json`, `--fire`, `--container-status`) and dir-based resolution; the fork grows one generic rail feature. (2) A one-time classification pass over `tasks.json` assigns gate edges (`Kill all` is container-context — host `lsof` would kill docker-proxy, not the servers; `Open browser to root dev` is host-shaped and its script sheds the VS Code browser helper). (3) Tabs whose shell runs *inside* the container may not show the hotbar (cwd tracking dependent) — explicitly out of scope for v1; firing is host-side regardless. (4) VS Code remains fully usable as a fallback surface: same tasks.json, same guard semantics, no fork lock-in for the task definitions themselves.
